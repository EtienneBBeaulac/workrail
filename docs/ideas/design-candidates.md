# Design Candidates: Temporal.io Pattern Adaptation for WorkRail Auto

Generated by wr.discovery workflow, 2026-04-14. Raw investigative material.

## Problem Understanding

**Core tensions:**
1. Applicability vs Fidelity: Temporal patterns exist for code-defined workflows; WorkRail workflows are JSON-declarative. Abstraction level must be chosen carefully.
2. MVP scope vs long-term correctness: namespace isolation, versioning behavior, inline triggers are all "right" long-term but add scope.
3. Engine purity vs trigger integration: inline trigger declarations in workflow JSON make the engine topology-aware (concern mixing risk).
4. Activity retry vs step model: Temporal retries activities independently; WorkRail's step model is coarser (no sub-step retry today).

**Likely seam:** src/daemon/ (new) + workflow.schema.json (3 optional fields) + src/trigger/ (new port).

**What makes this hard:**
Finding the right abstraction level at which Temporal patterns transfer. At code level, almost nothing transfers. At conceptual level, almost everything transfers. The value is in the middle: specific mechanisms (buffer/dispatch, PINNED/AUTO_UPGRADE semantics, namespace isolation contract) adaptable without importing the full implementation.

## Philosophy Constraints

From /Users/etienneb/CLAUDE.md:
- "Architectural fixes over patches" -> version adapters preferred over patched()-style guards
- "Make illegal states unrepresentable" -> discriminated union VersioningBehavior enum, typed trigger types
- "Immutability by default" -> daemon state via event log, not mutable session objects
- "DI for boundaries" -> LLM client and trigger dispatcher as injected ports

Philosophy conflict: "Architectural fixes" vs "YAGNI with discipline" -- namespace isolation is architecturally correct but may be premature. Resolution: orgId as DI arg defaulting to "default" satisfies both.

## Impact Surface

- workflow.schema.json (3 optional fields + defaults)
- src/v2/infra/local/session-lock/index.ts (workerId -> orgId extension)
- src/trigger/ (new TriggerDispatcher port)
- src/daemon/ (new daemon module)
- All bundled workflows: unaffected (all new fields have defaults)

## Candidates

### Candidate 1: Minimal Adaptation (MVP-first)
**Summary:** 4 atomic patterns in src/daemon/ only, zero schema changes.
**Tensions resolved:** MVP scope strictly. Defers schema correctness.
**Tensions accepted:** versioningBehavior lives in daemon code only (leaky abstraction).
**Boundary:** src/daemon/ only. Pure extension.
**Failure mode:** Retrofit cost at v2 when schema changes needed mid-production.
**Repo pattern:** Follows -- V2Dependencies DI extension.
**Gain:** Fastest MVP. No schema churn.
**Give up:** Explicit versioning type at workflow definition level.
**Scope:** Best-fit for MVP, too narrow for 6-month horizon.
**Philosophy:** Honors YAGNI. Conflicts with "make illegal states unrepresentable."

### Candidate 2: Adopt + Schema (Recommended with modification)
**Summary:** Add versioningBehavior + orgId DI now; defer triggers[] to dedicated sprint.
**Tensions resolved:** Applicability vs fidelity (explicit types). Architectural correctness.
**Tensions accepted:** Small additional schema work (3 lines in schema, all optional).
**Boundary:** workflow.schema.json + src/daemon/ + src/trigger/ interface only.
**Failure mode:** Trigger schema may need iteration -- separating from this sprint mitigates.
**Repo pattern:** Adapts -- extends workflow.schema.json following existing optional-field pattern (see rigorMode, automationLevel etc. in bundled workflows).
**Gain:** Explicit VersioningBehavior discriminated union. orgId seam for multi-tenancy.
**Give up:** Inline trigger declarations (deferred, not lost).
**Scope:** Best-fit. All 3 new fields have safe defaults; no bundled workflow changes required.
**Philosophy:** Honors "make illegal states unrepresentable," "architectural fixes over patches," DI for boundaries.

### Candidate 3: WorkRail-native YAGNI
**Summary:** Build only what MVP needs, no Temporal/Prefect pattern influence.
**Tensions resolved:** YAGNI strictly.
**Tensions accepted:** Known retrofit cost at v2 (versioningBehavior, orgId, triggers).
**Boundary:** src/daemon/ only.
**Failure mode:** Simultaneous retrofit of session store + versioning + triggers at v2 = expensive.
**Repo pattern:** Follows strictly -- pure additive.
**Gain:** Fastest MVP. Zero decisions made prematurely.
**Give up:** Architectural correctness on 3 design decisions that are cheap to get right now.
**Philosophy:** Honors YAGNI. Conflicts with "architectural fixes over patches."

### Candidate 4: Temporal as Backend (Rejected)
**Summary:** Use Temporal server as WorkRail's durable state backend.
**Rejected because:** Breaks portability guarantee (npx -y @exaudeus/workrail). WorkRail's session model is already superior. Self-defeating for positioning.
**When to revisit:** WorkRail Auto cloud at scale (100+ orgs). Temporal's CQRS + multi-tenant battle-testing would matter there.

## Comparison and Recommendation

**Recommended:** Candidate 2 with modification -- ship versioningBehavior + orgId DI now, defer triggers[] schema.

**Rationale:**
- versioningBehavior: cheap schema addition (1 optional field, default "PINNED"), removes leaky abstraction from C1, satisfies "make illegal states unrepresentable"
- orgId DI: LocalSessionLockV2 workerId fix (already in backlog) is the natural extension point; orgId = "default" is transparent to all existing code
- Deferred triggers: inline trigger declarations are directionally right (Prefect BaseDeploymentTrigger is the model) but the exact schema needs prototyping; defer to avoid premature commitment
- Buffer/dispatch for human-in-loop: REST poll (POST /api/v2/sessions/:id/resume) is sufficient for MVP; event-push is the Temporal Activator model but adds complexity not needed at MVP scale

## Self-Critique

**Strongest counter-argument:** versioningBehavior doesn't matter until there are production workflows AND engine upgrades -- there are none yet. YAGNI says don't add it. (Counter: the daemon's workflow loader needs to implement PINNED semantics even for test runs; the schema field makes an already-implemented invariant explicit, not speculative.)

**Narrower option:** C1 would work for daemon MVP. Loses because versioningBehavior lives only in daemon code, invisible to workflow definitions -- leaky abstraction.

**Broader option:** Full C2 with triggers. Justified when trigger schema is prototyped and validated.

**Invalidating assumption:** If versioningBehavior="PINNED" is not the right default. Temporal recommends PINNED for production -- WorkRail should match this.
