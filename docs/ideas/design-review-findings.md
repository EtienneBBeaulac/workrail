## Runner-Up / Simpler Alternative Review

Runner-up (C1+orgId) is a real fallback. It loses only on: versioningBehavior leaky abstraction (behavior implicit in daemon, not explicit in workflow definition). The hybrid of C1 + orgId path change is viable but weaker than the selected direction.

Simpler variant: C1+orgId (no schema changes). Satisfies acceptance criteria but accepts the leaky abstraction. Genuinely viable if daemon sprint timeline is at risk.

## Findings

**YELLOW (informational):** versioningBehavior field will be documentation-only until the daemon is implemented. This is acceptable but should be called out in the backlog item ("schema field added in X sprint, daemon reads it in Y sprint").

**YELLOW (informational):** The startup migration for orgId path change (~/.workrail/sessions/<sessionId>/ -> ~/.workrail/sessions/default/<sessionId>/) needs to be a zero-data-loss atomic move. This is a 10-line migration function, not a risk.

**GREEN:** No Red or Orange findings. The selected direction is sound.

## Recommended Revisions

1. Drop `requiredOrgId` from workflow schema (deployment-time concern, not author concern). Already incorporated in selected direction.
2. Add one-time startup migration for session store path (orgId prefix). Already incorporated.
3. Document in the daemon backlog item that `versioningBehavior` schema field is added in the schema sprint, read in the daemon sprint (two separate items, explicit dependency).

## Residual Concerns

1. The trigger system design (Prefect BaseDeploymentTrigger model) needs prototyping before the triggers[] schema sprint. The exact schema shape (how webhook + cron + event triggers are expressed in JSON) has not been validated against real workflow authors' mental models.
2. The REST control plane endpoint (POST /api/v2/sessions/:id/resume) for human-in-loop gates needs implementation before the daemon can ship a complete human-in-loop story. This is a dependency but not a blocker for the daemon loop itself.
