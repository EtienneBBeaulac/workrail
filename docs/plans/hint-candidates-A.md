# Artifact Submission Hint Candidates -- Engine-Anchored Angle

**Status:** Design candidates (2026-05-20)
**Angle:** Fix at source -- engine layer owns the error message at the point where the compiler
already knows exactly what went wrong. No cortex patches, no runtime layers.

---

## Problem Restatement (for candidates)

Three failure surfaces, all in the engine layer:

1. `reasonToBlocker()` in `reason-model.ts` -- the `missing_required_output` and
   `invalid_required_output` cases dispatch on `ASSESSMENT_CONTRACT_REF` and hardcode
   `notesMarkdown` advice for every other contractRef. Wrong for all artifact contracts.

2. `blocked_attempt_limit_exceeded` in `advance.ts` (line 137) -- hardcodes
   `wr.assessment` in the error message regardless of what contract is actually required.
   The `contractRef` is available via `nodeCreated.data.snapshotRef` -> snapshot -> blockers.

3. Wrong-kind artifact detection -- `validateArtifactContract()` filters by kind before
   checking whether any artifact with the right shape exists. When the agent submits
   `wr.assessment` where `wr.loop_control` is required, the error is `MISSING_REQUIRED_ARTIFACT`
   with no mention of what was submitted. The agent is told nothing was provided.

The engine has all the information: `contractRef`, the submitted artifacts array, and each
schema's `getBlockedMessage()` function (already registered in `CONTRACT_BLOCKED_MESSAGES`
in `prompt-renderer.ts`). These candidates differ in how fully that information is used.

## Approach 1 (p=0.85): Dispatch suggestedFix Through getBlockedMessage in reasonToBlocker

**Primary mechanism:** Replace the binary `ASSESSMENT_CONTRACT_REF` dispatch in
`reasonToBlocker()` with a call to `CONTRACT_BLOCKED_MESSAGES[contractRef]()`, making each
artifact schema the single source of its own fix message.

### Tension resolution

- **Fix at source vs. cortex layer:** Pure engine fix -- `reason-model.ts` is the
  origin of the bad `suggestedFix`. The cortex layer is untouched.
- **Correctness vs. completeness:** Correctness-only. Fixes the wrong-field advice and the
  missing scaffold. Does not address wrong-kind detection (Case D).
- **Which guidance source is authoritative:** The blocker's `suggestedFix` field becomes
  authoritative because it is generated by the engine at the point of failure, adjacent to
  the current turn in the agent's context window.
- **New contractRef extensibility:** Adding a new artifact contract requires only adding
  `getBlockedMessage()` to the new schema file. `reasonToBlocker()` never needs to change.

### Key tradeoffs

- `reason-model.ts` currently imports only `ASSESSMENT_CONTRACT_REF` from
  `schemas/artifacts/index.js`. Importing `CONTRACT_BLOCKED_MESSAGES` from
  `prompt-renderer.ts` would create a cross-domain dependency: `domain/ -> domain/`.
  The cleaner option is to extract `CONTRACT_BLOCKED_MESSAGES` into a new file
  `schemas/artifacts/blocked-messages.ts` that both `reason-model.ts` and
  `prompt-renderer.ts` import from. This is a two-file addition with no behavior change
  outside the two fix sites.
- The `suggestedFix` field has a 1024-byte budget. `getBlockedMessage()` returns a
  `string[]` -- joining with `\n` typically fits within 300-500 bytes for any current
  contract, well inside budget.
- Does NOT fix `blocked_attempt_limit_exceeded` (separate function, separate file).
  That is approach 2 scope.

### Implementation sketch

Files changed:

- `src/v2/durable-core/schemas/artifacts/blocked-messages.ts` (new) -- extract
  `CONTRACT_BLOCKED_MESSAGES` record from `prompt-renderer.ts` and export it here.
  Add `ASSESSMENT_CONTRACT_REF` entry pointing at a `getAssessmentBlockedMessage()`
  function added to `assessment.ts`.

- `src/v2/durable-core/schemas/artifacts/assessment.ts` (new export) -- add
  `getBlockedMessage(): readonly string[]` following the same pattern as `loop-control.ts`.
  Returns: artifact contract ref, kind discriminant, required fields, canonical format
  with `assessmentId` and at least one dimension filled in.

- `src/v2/durable-core/domain/prompt-renderer.ts` -- change `CONTRACT_BLOCKED_MESSAGES`
  from a local const to a re-export from `blocked-messages.ts`. One import change.

- `src/v2/durable-core/domain/reason-model.ts` -- in `reasonToBlocker()`, replace the
  `missing_required_output` and `invalid_required_output` branch bodies:

  ```typescript
  // Before:
  suggestedFix:
    contractRef === ASSESSMENT_CONTRACT_REF
      ? 'Call continue_workflow without output to rehydrate...'
      : 'Call continue_workflow without output to rehydrate the current step, then retry with output.notesMarkdown...',

  // After:
  suggestedFix: CONTRACT_BLOCKED_MESSAGES[contractRef]?.()?.join('\n') ??
    `Provide a valid artifact for contractRef=${contractRef}. Rehydrate with continue_workflow (no output) then retry.`,
  ```

  The `invalid_required_output` case gets the same treatment, prepending
  "Invalid artifact submitted -- correct the following and retry:\n".

### Quality-aspirational criterion

Partially satisfied. The correct scaffold appears in the blocked tool result (highest
salience position). But Case D (wrong-kind submitted) still produces a generic
`MISSING_REQUIRED_ARTIFACT` message with no mention of what was submitted. The agent
may need a second hint cycle unless the scaffold alone is sufficient.

## Approach 2 (p=0.55): Fix blocked_attempt_limit_exceeded to Use Actual contractRef

**Primary mechanism:** At the circuit-breaker site in `advance.ts`, read the actual
`contractRef` from the blocked snapshot's blocker report rather than hardcoding
`"wr.assessment"`.

### Tension resolution

- **Fix at source vs. cortex layer:** Engine fix, same file as the bug, one level up from
  Approach 1. The `contractRef` is available via the blocker report already stored in the
  blocked snapshot.
- **Correctness vs. completeness:** Correctness-only. Fixes the hardcoded wrong type. Does
  not change the scaffold quality beyond using the correct type name.
- **Which guidance source is authoritative:** The `blocked_attempt_limit_exceeded` error
  is an `InternalError` returned as a PRECONDITION_FAILED MCP response, not a normal
  blocked tool result. It is at the call site of `continue_workflow` / `complete_step`.
- **New contractRef extensibility:** Once fixed, no code change needed for new contracts --
  `contractRef` is read from the persisted snapshot data.

### Key tradeoffs

- `advance.ts` already loads the blocked snapshot (`snap`) before the circuit-breaker
  check. The snapshot contains `engineState.blocked.blockers`. The first blocker with
  `pointer.kind === 'output_contract'` yields the actual `contractRef`.
- This is a targeted one-function fix: `advanceAndRecord()` in `advance.ts`, the
  `chainDepth >= MAX_BLOCKED_ATTEMPT_RETRIES` branch.
- Separating this from Approach 1 means two separate PRs or a combined PR. These two fixes
  are orthogonal (different failure surfaces) so they can be applied independently.
- Without Approach 1, the circuit-breaker message would show the right `contractRef` but
  the regular blocked message would still say `notesMarkdown`. Both should be fixed
  together for consistency, but each is independently valuable.

### Implementation sketch

Files changed:

- `src/mcp/handlers/v2-execution/advance.ts` -- in the
  `chainDepth >= MAX_BLOCKED_ATTEMPT_RETRIES` branch:

  ```typescript
  // Before:
  message: `Assessment gate failed after ${MAX_BLOCKED_ATTEMPT_RETRIES} attempts. ` +
    `Submit a valid wr.assessment artifact. Required format:\n` +
    `\`\`\`json\n` +
    `{ "artifacts": [{ "kind": "wr.assessment", ... }] }\n\`\`\``,

  // After:
  const firstOutputBlocker = blocked.blockers.blockers
    .find(b => b.pointer.kind === 'output_contract');
  const contractRef = firstOutputBlocker?.pointer.contractRef ?? 'unknown';
  const scaffold = CONTRACT_BLOCKED_MESSAGES[contractRef]?.()?.join('\n') ??
    `Provide a valid artifact for contractRef=${contractRef}.`;

  message: `Step blocked after ${MAX_BLOCKED_ATTEMPT_RETRIES} consecutive retries on ` +
    `contractRef=${contractRef}. Circuit breaker fired.\n\n` +
    scaffold,
  ```

  Import `CONTRACT_BLOCKED_MESSAGES` from `schemas/artifacts/blocked-messages.ts`
  (the new shared location from Approach 1, or inline for now as a stopgap).

### Quality-aspirational criterion

Not fully satisfied alone. This only fires after 3 failures, by which point the agent is
likely stuck in an attention-degraded state. The quality-aspirational criterion requires
correct guidance on the FIRST failure. Approach 2 is a necessary fix but not sufficient.
When combined with Approach 1, the combination achieves the criterion.

## Approach 3 (p=0.35): Wrong-Kind Detection with Diagnostic Feedback

**Primary mechanism:** In each per-contract validation function inside
`artifact-contract-validator.ts`, detect artifacts of any non-matching kind before
reporting `MISSING_REQUIRED_ARTIFACT`, and if found, report
`WRONG_KIND_ARTIFACT_SUBMITTED` (a new error code) naming what was submitted vs. what
was required.

### Tension resolution

- **Fix at source vs. cortex layer:** Still engine layer, but a different boundary than
  Approaches 1-2. This is the artifact validation boundary where the engine currently
  silently discards wrong-kind artifacts. It is the most precise source for Case D.
- **Correctness vs. completeness:** The most complete fix for Case D. Addresses the
  observation that the validator knows exactly what the agent submitted, but does not
  surface it.
- **Which guidance source is authoritative:** The `MISSING_REQUIRED_ARTIFACT` message
  in the blocker becomes `WRONG_KIND_ARTIFACT_SUBMITTED` (or the message is enriched
  inline). The blocker tool result is the authoritative source for Case D.
- **New contractRef extensibility:** Requires one change per new contract schema file
  (add wrong-kind detection). More maintenance than Approaches 1-2. Alternatively, a
  single cross-kind scan in `validateArtifactContract()` could detect any submitted-but-
  wrong-kind artifact by checking against the known kind registry without touching each
  schema file.

### Key tradeoffs

- Touching `artifact-contract-validator.ts` is more invasive than touching
  `reason-model.ts`. The validator is called at the boundary between agent output and
  engine state; the contract here is that it returns a discriminated union of known error
  types. Adding a new error code (`WRONG_KIND_ARTIFACT_SUBMITTED`) requires updating
  `ArtifactContractValidationError`, `formatArtifactValidationError()`, the callers
  in `validation-criteria-validator.ts`, and the blocker construction path.
- Alternative (lower-surface): do the wrong-kind scan in `validateArtifactContract()`
  itself before dispatching to per-contract validators. Extract submitted kinds from the
  artifacts array, compare against the expected kind for `contractRef`, and include the
  submitted kinds in the `MISSING_REQUIRED_ARTIFACT` message. This requires no new error
  code and no change to callers -- just enriches the existing error message string.
- Without Approaches 1-2, the enriched wrong-kind message would still be embedded inside
  the existing (misleading) `suggestedFix`. Adding this fix alone without the others
  makes Case D better but leaves Cases B and C wrong.

### Implementation sketch

Files changed:

- `src/v2/durable-core/domain/artifact-contract-validator.ts` -- in
  `validateArtifactContract()`, before dispatching to per-contract validators:

  ```typescript
  // After isValidContractRef() check, before dispatch:
  const expectedKind = CONTRACT_KIND_BY_REF[contractRef]; // new lookup map
  const submittedKinds = artifacts
    .filter(a => typeof a === 'object' && a !== null)
    .map(a => (a as Record<string, unknown>).kind)
    .filter((k): k is string => typeof k === 'string');

  const hasWrongKind = submittedKinds.some(k => k !== expectedKind && k.startsWith('wr.'));
  ```

  Then in each per-contract validator's `MISSING_REQUIRED_ARTIFACT` path, enrich the
  message:

  ```typescript
  message: hasWrongKind
    ? `Wrong artifact kind submitted (got: ${submittedKinds.join(', ')}; expected kind='${expectedKind}' for ${contractRef}).`
    : `Required artifact missing: ${contractRef}. Agent must provide an artifact with kind='${expectedKind}'.`,
  ```

- Add `CONTRACT_KIND_BY_REF: Readonly<Record<ArtifactContractRef, string>>` to
  `schemas/artifacts/index.ts`, mapping each contractRef to its kind literal
  (e.g., `'wr.contracts.loop_control' -> 'wr.loop_control'`).

### Quality-aspirational criterion

Satisfies Case D alone. When combined with Approaches 1-2, the combination fully
satisfies the quality-aspirational criterion: each failure mode produces a precise,
actionable, non-redundant first hint that leads directly to correction on the next
attempt.

## Approach 4 (p=0.07): Runtime-Derived suggestedFix from Zod Schema Introspection

**Primary mechanism:** At the point of failure in `reasonToBlocker()`, derive a minimal
valid instance scaffold by introspecting the Zod schema shape for the required contract,
rather than delegating to hand-authored `getBlockedMessage()` functions. The scaffold
is always structurally valid and reflects the actual current schema.

### Tension resolution

- **Fix at source vs. cortex layer:** Engine fix, but at a new level: the Zod schema
  itself becomes the source of truth for the example rather than a maintained prose
  string. This is philosophically purer -- the schema IS the spec.
- **Correctness vs. completeness:** Potentially highest correctness: the derived example
  always reflects the actual schema. But completeness is limited -- Zod introspection
  cannot produce semantically meaningful field values (it can only produce type-correct
  minimums). `rationale: ""` is a valid Zod string but useless guidance.
- **Which guidance source is authoritative:** The schema itself, via programmatic
  derivation. This removes the `getBlockedMessage()` maintenance burden entirely.
- **New contractRef extensibility:** Zero changes needed. A new Zod schema automatically
  produces a derived example.

### Key tradeoffs

- Zod does not have a public `generateMinimalInstance()` API. Implementing a recursive
  schema walker for `.object()`, `.literal()`, `.enum()`, `.array()`, `.string()`,
  `.number()`, `.optional()`, `.union()` is ~100-200 lines of non-trivial code and
  requires thorough testing.
- The derived example will always use the first enum value and an empty string for
  required strings with no `.min()` hint -- this is not necessarily the most instructive
  example. Hand-authored `getBlockedMessage()` examples use semantically meaningful
  values (e.g., `"decision": "stop"`, `"verdict": "approved"`).
- Strict schemas (`.strict()`) will validate fine but the derived example may include
  only mandatory fields, which is actually ideal for showing minimum requirements.
- This approach is architecturally cleaner but empirically riskier -- the quality of
  guidance is now governed by schema shape, not by the author's intent. A schema that
  uses `.string()` everywhere produces an unhelpful scaffold.
- For the immediate bug fix, this is over-engineered. As a long-term foundation for
  zero-maintenance scaffolding it is compelling, but the maintenance cost is shifted from
  `getBlockedMessage()` text to the Zod walker code.

### Implementation sketch

Files changed:

- `src/v2/durable-core/schemas/artifacts/minimal-instance-generator.ts` (new) --
  Zod schema walker that produces a minimal valid instance:

  ```typescript
  function deriveMinimalInstance(schema: z.ZodTypeAny): unknown { ... }
  ```

  Handles: `ZodObject`, `ZodLiteral`, `ZodEnum`, `ZodString`, `ZodNumber`,
  `ZodBoolean`, `ZodArray`, `ZodOptional` (omits), `ZodUnion` (first variant).

- `src/v2/durable-core/domain/reason-model.ts` -- replace `getBlockedMessage()` lookup
  with `deriveMinimalInstance(CONTRACT_SCHEMAS[contractRef])`, JSON-serialize the result,
  and format as a code block.

- `src/v2/durable-core/schemas/artifacts/index.ts` -- export
  `CONTRACT_SCHEMAS: Readonly<Record<ArtifactContractRef, z.ZodTypeAny>>` mapping each
  ref to its schema object.

### Quality-aspirational criterion

Satisfies it in theory for any schema that produces a meaningful minimal instance.
In practice, semantically empty field values reduce actionability. A hybrid approach
(derive structure, annotate with `getBlockedMessage()` comments) would be strictly better
but is also more complex. This approach is listed as a candidate to represent the
"schema is the spec" end of the design space, not as the recommended path.

## Approach 5 (p=0.04): Unified Hint Builder Injectable at Advance Time

**Primary mechanism:** Extract all hint-construction logic (suggestedFix text,
wrong-kind detection, circuit-breaker message) into a single pure function
`buildArtifactHint(contractRef, submittedArtifacts): string` that is called once from
a shared site -- the `buildBlockerReport()` call or the `buildBlockedOutcome()` path --
and injected into every blocker that has `pointer.kind === 'output_contract'`. The hint
builder has access to both the `contractRef` and the full submitted artifacts array,
making it the only place that ever needs to change when hint quality improves.

### Tension resolution

- **Fix at source vs. cortex layer:** Engine fix, but at a higher abstraction level
  than Approaches 1-3. Rather than fixing three separate sites, this approach creates
  a new abstraction (the hint builder) that owns all hint logic. The three bugs become
  the first tests against the new abstraction.
- **Correctness vs. completeness:** Highest completeness -- the hint builder handles
  missing artifact, wrong kind, and invalid schema in one unified path.
- **Which guidance source is authoritative:** The `buildArtifactHint()` function,
  which delegates to `getBlockedMessage()` per contract and enriches with submitted-
  artifact context. One source, one place to update.
- **New contractRef extensibility:** One change needed: add `getBlockedMessage()` to
  the new schema and register it. The hint builder itself requires no change because
  it delegates to the per-contract function.

### Key tradeoffs

- This approach requires wiring `submittedArtifacts` through to `buildBlockerReport()`
  or deeper into the blocked outcome path. Currently `buildBlockerReport()` takes only
  `ReasonV1[]` with no artifact context. Adding artifact context to the blocker build
  path requires either: (a) threading `artifacts` as a new optional parameter, or
  (b) enriching `missing_required_output` and `invalid_required_output` variants of
  `ReasonV1` to carry submitted artifacts.
  
  Option (b) is architecturally cleaner (the reason carries its own diagnostic context)
  but requires updating all call sites that construct these reason variants. Currently
  there are 3-4 call sites, all in the advance path.

- The circuit-breaker fix (Approach 2) and the regular blocked fix (Approach 1) share
  the same hint builder, eliminating the duplication currently present between them.

- This is the highest-surface fix but also the one that produces the cleanest
  steady-state architecture: a single authoritative hint function replaces three
  scattered hardcoded strings with no shared structure.

- Risk: threading `artifacts` through the reason model may touch load-bearing
  infrastructure (`reason-model.ts` is noted in AGENTS.md as part of `src/v2/`).
  Changes here require careful testing against the contract tests.

### Implementation sketch

Files changed:

- `src/v2/durable-core/domain/artifact-hint-builder.ts` (new) -- pure function:

  ```typescript
  export function buildArtifactHint(
    contractRef: string,
    submittedArtifacts: readonly unknown[],
  ): string {
    const submittedKinds = submittedArtifacts
      .filter(a => typeof a === 'object' && a !== null)
      .map(a => (a as Record<string, unknown>).kind)
      .filter((k): k is string => typeof k === 'string' && k.startsWith('wr.'));

    const getMsg = CONTRACT_BLOCKED_MESSAGES[contractRef];
    const canonical = getMsg?.()?.join('\n') ?? `Provide a valid artifact for contractRef=${contractRef}.`;

    if (submittedKinds.length > 0) {
      const wrongKinds = submittedKinds.filter(k => k !== CONTRACT_KIND_BY_REF[contractRef]);
      if (wrongKinds.length > 0) {
        return `Wrong artifact kind submitted (got: ${wrongKinds.join(', ')}).\n\nExpected:\n${canonical}`;
      }
    }

    return canonical;
  }
  ```

- `src/v2/durable-core/domain/reason-model.ts` -- `reasonToBlocker()` for
  `missing_required_output` and `invalid_required_output`: replace `suggestedFix`
  hardcoded strings with `buildArtifactHint(contractRef, reason.submittedArtifacts ?? [])`.
  
  Add `submittedArtifacts?: readonly unknown[]` to `missing_required_output` and
  `invalid_required_output` variants of `ReasonV1`.

- `src/v2/durable-core/domain/blocking-decision.ts` -- pass the submitted artifacts
  array when constructing `missing_required_output` and `invalid_required_output` reasons.

- `src/mcp/handlers/v2-execution/advance.ts` -- circuit-breaker site: replace
  hardcoded scaffold with `buildArtifactHint(contractRef, [])` (no submitted artifacts
  available at this site, so the hint is schema-only).

### Quality-aspirational criterion

Fully satisfied. Every failure mode (missing, wrong-kind, invalid, circuit-breaker)
produces a precise, correct, non-redundant first hint derived from the exact failure
context. The engine is the single authoritative source. No duplication, no static
strings at multiple sites.

## Comparison Matrix

| Criterion | Approach 1 | Approach 2 | Approach 3 | Approach 4 | Approach 5 |
|---|---|---|---|---|---|
| All guidance sources consistent | Fixes 1 of 3 | Fixes 1 of 3 | Doesn't fix 1 | Fixes 1 of 3 | Fixes all 3 |
| New contractRef: at most 1 change | Yes (add `getBlockedMessage()`) | No change | Add entry to kind map | Zero changes | Add `getBlockedMessage()` |
| Wrong-kind names what was submitted | No | No | Yes | No | Yes |
| Quality-aspirational: first hint = correct | Partially | No (fires after 3) | Partially | Theoretically | Yes |
| Vision-aligned: all entry points | Yes | Yes | Yes | Yes | Yes |
| Surface area of change | Low | Low | Medium | High | Medium-high |
| Risk to load-bearing infra | Low | Low | Low | Medium | Medium |

## Recommended Progression

Apply these in order. Each is independently correct and non-breaking:

1. Approach 1 (high probability, low risk, immediate correctness fix for Cases B and C)
2. Approach 2 (medium probability, low risk, fixes the circuit-breaker wrong-type)
3. Approach 3 (lower probability but addresses Case D which is silent and hard to debug)

Approaches 1-3 combined achieve approximately 85% of the quality-aspirational criterion
with minimal infrastructure change. Approach 5 achieves 100% but requires threading
artifacts through the reason model and is a more appropriate Phase 2 investment.
Approach 4 is a long-term architectural aspiration, not a near-term fix.
